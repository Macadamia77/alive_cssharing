# Instagram / Facebook 콘텐츠 규칙

## 카드뉴스
- 카드 수는 방향의 story_plan 수에 따라 총 5장 또는 6장으로 작성한다.
- 중간 카드에 필요한 서로 다른 메시지가 3개면 후킹 1장, 중간 카드 3장, CTA 1장으로 총 5장으로 구성한다.
- 중간 카드에 필요한 서로 다른 메시지가 4개면 후킹 1장, 중간 카드 4장, CTA 1장으로 총 6장으로 구성한다.
- 카드 수를 늘리기 위해 같은 내용을 표현만 바꿔 반복하지 않는다.
- 첫 장 후킹은 05-hook-patterns.md의 7가지 유형(통계·수치 제시형, 반문형, 자기고백/공감형, 경고·손실회피형, 비교·재정의형, 대상 지정형, FOMO·트렌드형) 중 이번 주제에 가장 잘 맞는 유형을 매번 새로 판단해서 고르는 것을 기본으로 한다.
- "~한데 ~인 이유", "~쌓이는데 ~막막한 이유"처럼 상황과 결론을 단순 나열만 하고 끝내는 안전한 문제 제기형 문장을 기본값으로 쓰지 않는다. 이 패턴은 7가지 유형이 전부 주제에 안 맞을 때만 최후 수단으로 쓴다.
- 자기고백/공감형을 쓸 때는 담당자가 스스로를 떠올리게 하는 상황·감정 묘사이되, 대상을 소비자·개인 셀러가 아닌 B2B CS 담당자로 유지한다.
  - 예: "쌓이기만 하는 VOC, 확인은 하고 계세요?", "그 문의, 이번 달에도 반복됐다면"
- 05-hook-patterns.md의 예시 문구를 그대로 베끼지 않고 실제 주제에 맞게 새로 쓴다.
- 마지막 장은 새로운 문제를 추가하지 않고, 서비스 확인 또는 운영 점검 CTA 하나로 마무리하는 것을 기본으로 한다.
- 참여 유도가 필요한 주제에서는 댓글 유도형 CTA도 사용할 수 있다.
  - 예: "댓글 남겨주시면 관련 자료를 DM으로 보내드려요"
  - 이때 제공하는 자료는 미리 준비된 고정 자료 하나여야 한다. "업종에 맞는", "귀사에 맞춤" 등 댓글마다 개별 제작이 필요한 표현은 쓰지 않는다.
- first와 CTA 카드의 강조색·스타일은 Figma 템플릿을 유지하므로 highlight_text를 비운다.
- 중간 카드 title은 반드시 2줄로 작성한다.
- title은 줄당 최대 15자 내외로 작성한다.
- 중간 카드 subtitle은 반드시 1줄이며 최대 25자 내외로 작성한다.
- title이 2줄일 때 줄바꿈은 글자 수를 맞추기 위해 임의 위치에서 끊지 않고, 주어/서술어 또는 수식어/명사 같은 의미 단위 경계에서 끊는다.
- 일반 박스 item.title은 최대 10자 내외로 작성한다.
- 일반 박스 item.body는 최대 25자 내외로 작성한다.
- 비교형 박스 item.title은 최대 8자, item.body는 최대 20자 내외로 작성한다.
- 문구가 길어질 경우 문장을 자르지 말고 처음부터 더 짧게 다시 작성한다.
- steps_vertical은 실제 순서, 단계, 진행 흐름이 있을 때만 사용한다.
- 하나의 카드뉴스 안에서 같은 layout_type은 최대 2회까지만 사용한다.
- 병렬 사례·조건·확인 항목은 steps_vertical 대신 stacked_boxes 또는 keyword_boxes를 우선 사용한다.

### 레이아웃 선택 규칙

* 레이아웃은 장식용으로 무작위 선택하지 않고, 카드 안 정보의 관계에 따라 선택한다.
* steps_vertical은 실제 순서, 점검 단계, 실행 단계가 있을 때만 사용한다.
* steps_vertical의 item.title에 "1.", "①", "1단계:"처럼 순서 번호를 직접 쓰지 않는다. 파란 원형 배지가 이미 순서 번호를 보여주므로 중복이다. title은 "축적", "분석"처럼 그 단계 자체를 가리키는 말로 시작한다.
* flow_process는 앞 단계가 다음 단계로 이어지는 실제 처리 흐름이 있을 때만 사용한다.
* compare_2col의 두 item은 하나씩 따로 읽었을 때도 "이건 안 좋은 쪽, 이건 좋은 쪽(또는 이건 A방식, 이건 B방식)"처럼 즉시 대비가 느껴져야 한다. 두 item을 하나씩 가려놓고 읽었을 때 둘 다 그냥 "각자의 사실 하나"로만 읽히고 서로 짝지어 대비되는 느낌이 없으면, compare_2col을 잘못 고른 것이다 — stacked_boxes나 keyword_boxes를 다시 검토한다.
  - 나쁜 예: "CS팀 내 소비" / "유관 부서 연결" — 각각 읽으면 그냥 사실 설명이라 둘이 왜 나란히 있는지 안 느껴진다.
  - 좋은 예: "CS팀만 소화" / "전 부서 공유" — "CS팀만"이라는 단어 자체에 이미 "부족하다"는 느낌이 있어서, 옆의 "전 부서"와 바로 대비된다.
* compare_2col은 두 대상, 두 기준, 이전과 이후처럼 정확히 2개를 비교할 때 사용한다.
* compare_2col은 기본적으로 "이전 vs 이후", "이렇게 하면 vs 저렇게 하면"처럼 나쁨/좋음이 갈리는 대비 구도로 쓴다고 가정한다. 이 경우 나쁜 쪽 item에 `tone: "bad"`, 좋은 쪽 item에 `tone: "good"`을 붙여 Figma·미리보기에서 X/O 배지로 구분되게 한다(07-figma-contract.md 참고).
* 예외적으로 두 항목이 대비가 아니라 "내부 CS팀 vs 외부 전담 파트너처럼 역할을 나눠 맡는 경우", "서로 다른 두 긍정적 대응 전략을 나열하는 경우"처럼 나쁨/좋음이 없는 병렬 나열이면 tone을 넣지 않는다. 이때는 compare_2col 대신 stacked_boxes 사용을 먼저 검토한다.
* 이 예외를 판단할 때 "두 항목이 각각 조직·주체 이름이니까 역할 분담"이라고 표면적으로만 보지 않는다. 카드의 title·subtitle이 이미 문제→해결 구도로 프레이밍돼 있다면(예: title "아무것도 안 바뀐다", subtitle "~에서만 소비되는 구조") 두 항목도 그 문제/해결에 대응하는 것이므로 tone을 넣는다.
  - 나쁜 예(tone 생략): title "리포트는 만드는데\n아무것도 안 바뀐다" / subtitle "VOC가 CS팀 내부에서만 소비되는 구조" 인데 items를 "CS팀 내 소비"(tone 없음) / "유관 부서 연결"(tone 없음)로 둠 → title이 이미 "안 바뀐다"는 문제를 제기했으므로 "CS팀 내 소비"는 tone: bad, "유관 부서 연결"은 tone: good이어야 한다.
  - tone을 생략해도 되는 경우는 카드 전체가 "이런 상황엔 이 방식, 저런 상황엔 저 방식"처럼 애초에 문제 제기 없이 옵션을 안내하는 카드일 때뿐이다.
* compare_2col의 item.title에 "OO 전/OO 후", "변화 전/변화 후", "이전/이후", "기존 방식/개선 방식"처럼 시간·상태만 가리키는 라벨을 쓰지 않는다. 이런 라벨은 카드 title이 이미 말하는 "달라진다"는 사실을 되풀이할 뿐 새 정보가 없어 밋밋하게 읽힌다. 대신 그 상태에서 실제로 누가 무엇을 하는지(또는 하지 않는지)를 title에 직접 담는다.
  - 나쁜 예: "구조 변화 전" / "구조 변화 후" (무엇이 바뀌는지 안 보임)
  - 좋은 예: "응대만 하고 종료" / "원인 찾아 전달" (실제 행동 차이가 보임), "CS팀만 소화" / "전 부서 공유" (누가 하는지가 보임)
* compare_2col을 "일반 CS 대행 vs CS쉐어링", "타사 vs CS쉐어링"처럼 경쟁사와 비교해 우리가 낫다는 식으로 쓰지 않는다("응대만 하고 끝내는 방식" vs "원인까지 추적하는 방식"처럼 방법론 차원으로 비교). 반면 "시간제 서비스 vs 쉐어링 서비스"처럼 우리 서비스의 두 옵션 중 상황별로 뭐가 맞는지 골라주는 비교는 괜찮다 — 이때는 둘 다 긍정적 옵션이므로 tone 없이 병렬로 쓴다(04-reference-examples.md 참고).
* keyword_boxes는 같은 축의 항목 3~4개를 병렬로 정리할 때 사용한다.
* stacked_boxes는 반복 신호, 예외 상황, 확인 항목처럼 독립된 정보 2~3개를 정리할 때 사용한다.
* 중간 카드가 3장인 경우 최소 2개 이상의 layout_type을 사용한다.
* 중간 카드가 4장인 경우 최소 3개 이상의 layout_type을 사용한다.
* 같은 layout_type을 연속된 중간 카드에 사용하지 않는다.
* 내용의 관계에 맞지 않는 레이아웃을 억지로 사용하지 않는다.
* 위 layout_type 다양성 규칙을 지키려고, **내용이 실제로는 같은 종류인데 레이아웃만 바꿔서 다른 카드처럼 보이게 만들지 않는다.** 레이아웃 다양성은 카드 내용이 실제로 다른 정보 관계(원인 나열 / 순서·단계 / 비교 / 도구·기능 병렬)를 가질 때 자연스럽게 나와야 하는 결과이지, 먼저 채워야 할 목표가 아니다.
  - 나쁜 예: 카드2 items "데이터 축적만" / "채널별 데이터 분리" / "유관 부서 연계 부족"(stacked_boxes)과 카드3 items "데이터만 보고" / "단순 보고서 취합" / "분산 정보"(keyword_boxes) — 두 카드가 사실상 같은 "원인 3가지"를 표현만 바꿔 반복하면서, 다양성 규칙 때문에 레이아웃만 억지로 바꾼 경우. 이러면 독자 입장에서 두 카드가 왜 다르게 생겼는지 이해가 안 된다.
  - 이런 상황이면 레이아웃을 바꾸지 말고, 애초에 두 카드를 하나로 합치거나(반복 금지 규칙 위반이므로) 카드3의 내용 자체를 카드2와 겹치지 않는 다른 축(예: 순서가 있는 해결 절차, 실제 비교)으로 다시 기획한다.

## Instagram / Facebook 캡션 규칙

- 카드 문구를 순서대로 복사하지 않고, 게시물 전체 관점과 점검 포인트를 요약한다.
- 모바일에서 읽기 쉽게 문단 사이 빈 줄을 사용한다.
- caption에는 이모지를 총 3~5개 사용한다.
- 첫 문장은 주제를 환기하는 한 줄로 시작하며, 문장 맨 앞에 💡, 📌, 🔎, ❓ 중 내용에 맞는 이모지 1개를 사용한다.
- 서비스 방식, 운영 기준, 비교 포인트처럼 한 번 더 강조할 내용이 있을 때는 📌를 1~2회 추가로 사용할 수 있다.
- CTA 문단에는 이모지를 반드시 넣을 필요는 없으며, 필요할 때만 👉를 사용한다.
- 😅, 😊, 🤔 등 감정 이모지도 문맥상 자연스러우면 사용할 수 있으나, 핵심 내용의 시작 표시는 ✅, ✔️, 📌을 우선 사용한다.
- 같은 이모지를 모든 문단에 반복하거나, 모든 문장 끝에 이모지를 붙이지 않는다.
- 해시태그는 caption에 넣지 않고 hashtags 배열에 분리한다.
- hashtags는 12~15개이며 #CS쉐어링 #CS대행 #고객센터대행을 포함한다.
- caption은 해시태그를 제외하고 약 450~650자 내외로 작성한다.
- caption은 도입 / 문제 맥락 / 핵심 포인트 2~3개 / CTA 순서로 작성한다.
- 각 문단은 빈 줄로 구분한다.
- 핵심 포인트 문단은 한 줄 요약으로 끝내지 말고, 왜 그런 문제가 생기는지 또는 담당자가 무엇을 확인해야 하는지 1~2문장으로 설명한다.

## 중간 카드 흐름 구조

- [선택된 방향]이 "감성 스토리텔링형"이면 이 섹션 대신 바로 아래 "스토리텔링형 카드 흐름"을 사용한다.
- 그 외(정보 전달형, 문제 해결형, 방향 미지정)에는 아래 패턴 중 주제 성격에 가장 자연스러운 것을 선택한다.
- 3장: 문제 현상 → 요인 분석 → 해결 방안 (기본)
- 4장 A: 문제 현상 → 요인 분석 → 해결 방안 → 해결 방안이 실제로 작동하는 방식 (해결책을 한 카드로 다 담기 부족할 때)
- 4장 B: 문제 현상 → 요인 분석 1 → 요인 분석 2 → 해결 방안 (요인이 복잡할 때)
- 4장 C: 문제 현상 → 요인 분석 → 해결 방안 1 → 해결 방안 2 (해결책이 2단계일 때)
- 각 카드 내 항목은 해당 카드의 축(현상/요인/해결책) 하나로 통일하며, 현상과 해결책을 한 카드에 섞지 않는다.
- 요인 분석 카드에서 해결 방안 카드로 넘어갈 때, 해결 방안 카드의 title은 앞 카드에서 다룬 요인을 해결하는 방향으로 자연스럽게 이어진다.
- 4장 A의 마지막 중간 카드("해결 방안이 실제로 작동하는 방식")는 "CS쉐어링이 다른 이유", "CS쉐어링이 그 구조를 만듭니다"처럼 브랜드를 주어로 내세우는 소개 카드로 쓰지 않는다. 여전히 담당자 입장에서 "이 방식이 실제로 어떻게 돌아가는지"를 설명하는 카드여야 하며, 브랜드명과 확인·문의를 유도하는 문구는 다음에 오는 CTA 카드에서만 등장시킨다.

## 스토리텔링형 카드 흐름

- "문제 현상 → 요인 분석 → 해결 방안"처럼 구조를 분해해서 설명하지 않는다. 대신 한 인물(또는 한 팀)의 시점을 처음부터 끝까지 따라가는 여정으로 카드를 구성한다.
- 3장: 도입 전 상황(어떤 어려움을 겪고 있었는지) → 변화의 계기(무엇을 시도했고 무엇이 달라지기 시작했는지) → 지금의 결과(무엇이 좋아졌는지)
- 4장: 도입 전 상황 → 변화의 계기 → 지금의 결과 → 다른 팀도 적용할 수 있는 이유(CS쉐어링 연결)
- 인물은 "OO 대표님", "OO팀장님"처럼 구체적 직함으로 설정하되, 실존 인물·실제 고객사가 아니라 가상의 예시임을 첫 카드 또는 body에서 명확히 드러낸다("가령", "한 B2B 쇼핑몰의 CS팀장님이라면"처럼). 이름을 붙이더라도 실명처럼 보이는 성+직함(예: 김 대리)은 피하고 직함 위주로 서술한다.
- 성과·수치는 "고객 만족도가 크게 올라갔다"처럼 막연하게 뭉개지 말고, 무엇이 어떻게 달라졌는지 구체적 상황으로 보여준다. 단, 검증되지 않은 구체적 %·건수를 실화처럼 단정하지 않는다.
- 각 카드의 items는 그 시점(도입 전/계기/결과)에서 실제로 있었던 구체적 상황·행동 2~3개로 채우며, 다른 시점의 내용을 섞지 않는다.
- CTA는 이 인물의 여정을 요약하지 않고, 읽는 사람이 자기 상황에 대입해볼 수 있는 질문이나 확인 행동으로 마무리한다.

## 문장 톤
- 도입은 가볍게 공감시키되, 문제 설명과 해석은 B2B 실무형으로 정리한다.
- 카드 title에는 이모지를 사용하지 않는다.
- item.title은 4~14자, item.body는 한두 줄로 읽히는 설명형 문장으로 작성한다.
- title, subtitle, body는 개념을 명사로 나열해 붙이지 않고, 주어와 서술어가 자연스럽게 이어지는 문장으로 작성한다.
  - 금지 예: 성수기 피로감에 더해 비수기 문의의 특성 (명사구+명사구 나열, 의미가 끊김)
  - 권장 예: 성수기 피로가 채 가시기도 전에 비수기 특유의 문의까지 겹치면서 (주어-서술어로 자연스럽게 연결)
- 문장을 소리 내어 읽었을 때 어색하면, 글자 수를 맞추기 위해 조사나 서술어를 생략하지 말고 표현 자체를 다시 쓴다.
- "~하는 이유", "~하지 않으려면?" 같은 문장 틀 자체는 문제가 아니다. 진짜 문제는 그 틀 안에 들어가는 **추상적인 서술어**다. "개선에 못 쓰는", "보고서에 멈추지"처럼 실제로 무슨 일이 일어나는지 그림이 안 그려지는 동사·표현을 쓰지 않는다. 그 카드의 items가 이미 구체적 내용을 담고 있다면, title도 그 수준으로 구체적인 동작·상태를 직접 서술한다.
  - 나쁜 예: "상담 데이터를 개선에 못 쓰는 이유" (어디에 "못 쓴다"는 건지 그림이 안 그려짐) / "VOC 분석, 보고서에 멈추지 않으려면?" ("멈춘다"가 무슨 상태인지 막연함)
  - 좋은 예: "상담 데이터, 쌓이기만 하고 그대로 묻힌다" / "VOC 분석은 보고서에서 끝나는 경우가 많다"
  - 한 세트(카드뉴스 한 편) 안에서 같은 문장 틀("~하는 이유"든 뭐든)을 3장 넘게 반복해서 쓰지 않는다 — 틀 하나를 계속 우려먹으면 그것대로 사람이 쓴 글처럼 안 읽힌다.
